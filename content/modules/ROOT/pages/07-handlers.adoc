= 1.6 - Handlers

Handlers are tasks that only run *when notified by another task*. They are ideal for service restarts that should happen only when something actually changes — for example, reloading a firewall after its rules are updated.

In this challenge, you will use Ansible Lightspeed to add Apache service management, firewall configuration, and a handler that reloads the firewall only when needed.

=====
.How handlers work
[%collapsible]
====
* A task uses `notify: <handler name>` to trigger a handler if it makes a change.
* Handlers are defined in a `handlers:` block at the same level as `tasks:`.
* A handler only runs *once per play*, even if it is notified by multiple tasks.
* If no notifying task reports a change, the handler never runs — keeping things idempotent.
====
=====

== Hands-On Experience

======
.☑️ Task 1 - Add service and firewall tasks with Lightspeed
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab and open *system_setup.yml*.
====

. First, add a new variable to the `vars:` section. After `package_name: httpd`, type:
+
[source,yaml]
----
    apache_service_name: httpd
----

. At the end of the existing tasks (before any `handlers:` section), use Lightspeed to generate each of the following tasks. Type the task name, press Enter, then Tab to accept:
+
[source,yaml]
----
    - name: Ensure Apache is running and enabled on web servers
----
+
[TIP]
====
*Expected:*
[source,yaml]
----
    - name: Ensure Apache is running and enabled on web servers
      ansible.builtin.service:
        name: "{{ apache_service_name }}"
        state: started
        enabled: true
      when: inventory_hostname in groups['web']
----
====

. Continue generating the next tasks:
+
[source,yaml]
----
    - name: Install firewalld on web servers
----
+
[source,yaml]
----
    - name: Ensure firewalld is running on web servers
----
+
[source,yaml]
----
    - name: Allow HTTP traffic on web servers
----
+
[TIP]
====
For the *Allow HTTP traffic* task, Lightspeed should generate a `notify:` line pointing to a handler. The expected output:

[source,yaml]
----
    - name: Allow HTTP traffic on web servers
      ansible.posix.firewalld:
        service: http
        permanent: true
        state: enabled
      when: inventory_hostname in groups['web']
      notify: Reload Firewall
----
====
=====
======


======
.☑️ Task 2 - Generate the handler with Lightspeed
[%collapsible]
=====
. After the last task, add a blank line and type:
+
[source,yaml]
----
  handlers:
----
. On the next line, type the handler name:
+
[source,yaml]
----
    - name: Reload Firewall
----
. Press *Enter*, then *Tab* to accept Lightspeed's suggestion.

[TIP]
====
*Expected handler output:*
[source,yaml]
----
  handlers:
    - name: Reload Firewall
      ansible.builtin.service:
        name: firewalld
        state: reloaded
----

The handler name *must match exactly* the string used in `notify:`. Lightspeed generates the correct module automatically.
====

. Save the file.

======
.☑️ Task 4 - Run again to see the handler stay quiet
[%collapsible]
=====
. Run the playbook a second time:
+
[source,shell]
----
ansible-navigator run system_setup.yml
----

[NOTE]
====
Because the firewall rule already exists, the *Allow HTTP traffic* task reports `ok` (no change) and does *not* notify the handler. The *Reload Firewall* handler never runs — the `PLAY RECAP` shows `changed=0`. This is handlers and idempotency working together.
