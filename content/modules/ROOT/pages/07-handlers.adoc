= 1.6 - Handlers

Handlers are tasks that only run *when notified by another task*. They are ideal for service restarts that should happen only when something actually changed — for example, reloading a firewall only after its rules are updated.

In this challenge, you will prompt Lightspeed to generate a playbook that adds Apache service management, firewall configuration, and a handler that reloads the firewall only when needed.

=====
.How handlers work
[%collapsible]
====
* A task uses `notify: <handler name>` to signal that something changed and the handler should run.
* Handlers are defined in a `handlers:` block at the same indentation level as `tasks:`.
* A handler only runs *once per play*, even if notified by multiple tasks.
* If no notifying task reports a change, the handler never runs — keeping the play idempotent.
====
=====

== Hands-On Experience

======
.☑️ Task 1 - Generate the playbook with service tasks and a handler
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

. Open the Lightspeed generation panel.
. Enter the following prompt:
+
----
Write a playbook targeting all hosts that creates a user named padawan, installs kernel security updates on web servers only, installs and starts httpd on web servers, installs and starts firewalld on web servers, allows HTTP traffic through the firewall on web servers and notifies a handler to reload firewalld, using variables for user_name padawan and package_name httpd and apache_service_name httpd
----
. Review the generated playbook in the new editor tab. Verify that:

* All existing tasks from previous modules are present
* `ansible.builtin.service` tasks start and enable both httpd and firewalld
* The `ansible.posix.firewalld` task has `notify: Reload Firewall`
* A `handlers:` block at the end contains the `Reload Firewall` handler

[TIP]
====
*Expected Lightspeed output:*

[source,yaml]
----
---
- name: Basic System Setup
  hosts: all
  become: true
  vars:
    user_name: 'padawan'
    package_name: httpd
    apache_service_name: httpd
  tasks:
    - name: Install security updates for the kernel
      ansible.builtin.dnf:
        name: 'kernel'
        state: latest
        security: true
        update_only: true
      when: inventory_hostname in groups['web']

    - name: Create a new user
      ansible.builtin.user:
        name: "{{ user_name }}"
        state: present
        create_home: true

    - name: Install Apache on web servers
      ansible.builtin.dnf:
        name: "{{ package_name }}"
        state: present
      when: inventory_hostname in groups['web']

    - name: Ensure Apache is running and enabled
      ansible.builtin.service:
        name: "{{ apache_service_name }}"
        state: started
        enabled: true
      when: inventory_hostname in groups['web']

    - name: Install firewalld
      ansible.builtin.dnf:
        name: firewalld
        state: present
      when: inventory_hostname in groups['web']

    - name: Ensure firewalld is running
      ansible.builtin.service:
        name: firewalld
        state: started
        enabled: true
      when: inventory_hostname in groups['web']

    - name: Allow HTTP traffic on web servers
      ansible.posix.firewalld:
        service: http
        permanent: true
        state: enabled
      when: inventory_hostname in groups['web']
      notify: Reload Firewall

  handlers:
    - name: Reload Firewall
      ansible.builtin.service:
        name: firewalld
        state: reloaded
----

The handler name in `notify:` must match the handler's `name:` exactly. Lightspeed generates this match automatically from the context of the prompt.
====

. Press `Ctrl+S` and save over *system_setup.yml* in the *ansible-files* directory.

====
Because the firewall rule already exists, the *Allow HTTP traffic* task reports `ok` (no change) and does *not* call `notify`. The *Reload Firewall* handler never runs. The `PLAY RECAP` shows `changed=0`. This is handlers and idempotency working together — configuration management without unnecessary service disruptions.

== Learning Outcomes

By completing this module, you now understand:

* How handlers run only when notified by a changed task — not on every play run
* Why `notify:` and `handlers:` names must match exactly
* How to combine conditionals and handlers to keep service restarts isolated to the correct hosts
* How to verify handler behavior by running the playbook twice
