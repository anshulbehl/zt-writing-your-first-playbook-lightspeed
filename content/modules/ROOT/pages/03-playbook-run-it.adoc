= 1.2 - Run the Playbook!

In this challenge, you will run the Lightspeed-generated playbook using *ansible-navigator*, verify that the user was created, and discover one of Ansible's most important concepts: *idempotency*.

== Hands-On Experience

======
.☑️ Task 1 - Run system_setup.yml
[%collapsible]
=====
[NOTE]
====
Switch to the *Control* tab.
====

. Change into the *ansible-files* directory:
+
[source,role="execute"]
----
cd /home/rhel/ansible-files
----

. Run the playbook:
+
[source,role="execute"]
----
ansible-navigator run system_setup.yml
----

[IMPORTANT]
====
The first run may take a minute or two. ansible-navigator pulls the Execution Environment container image and then runs the kernel update task on *node1*. Subsequent runs are much faster.
====

. Watch the output. The *PLAY RECAP* at the end summarizes changes made across all tasks:
+
image::navigator-output.png[ansible-navigator run output showing task results and PLAY RECAP,link=self,window=blank,width=700,title="Playbook run output"]

[NOTE]
====
The first task in every run is *Gathering Facts*. Ansible automatically collects system information from each managed node — hostname, OS version, IP addresses, installed packages — before running any tasks. Lightspeed uses this fact data to generate context-aware suggestions.

.☑️ Task 3 - Discover idempotency
[%collapsible]
=====
=====
.What is idempotency?
[%collapsible]
====
*Idempotency* means that running an operation once produces the same result as running it many times. An idempotent Ansible task only makes a change when the desired state does not already exist. If the state is already correct, the task reports `ok` and does nothing.

This is what separates Ansible from shell scripts: you can safely run a playbook repeatedly and it will never break a system that is already configured correctly.
====
=====

. Run the playbook a second time:
+
[source,role="execute"]
----
ansible-navigator run system_setup.yml
----

. Compare the output to the first run. Notice:

* Task status changed from *changed* (yellow) to *ok* (green)
* The *PLAY RECAP* shows `changed=0` — Ansible confirmed everything was already in place
+
image::idem-output.png[Second run showing ok status and changed=0 in PLAY RECAP,link=self,window=blank,width=700,title="Idempotent second run"]

. Switch to the *VS Code* tab, open *system_setup.yml*, select all the content, and use *Ansible Lightspeed → Explain*. Ask: "Why is this playbook idempotent?" and read the explanation.
