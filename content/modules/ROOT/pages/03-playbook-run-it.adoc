= 1.2 - Run and Verify the Playbook

In this challenge, you will run the comprehensive playbook generated in the previous module using *ansible-navigator*, verify that every component worked correctly across multiple hosts, and discover one of Ansible's most important concepts: *idempotency*.

== Hands-On Experience

======
.☑️ Task 1 - Run system_setup.yml
[%collapsible]
=====
[NOTE]
====
Switch to the *Control* tab.
====

. Change into the *ansible-files* directory:
+
[source,role="execute"]
----
cd /home/rhel/ansible-files
----

. Run the playbook:
+
[source,role="execute"]
----
ansible-navigator run system_setup.yml
----

[IMPORTANT]
====
The first run may take a few minutes. ansible-navigator pulls the Execution Environment container image and then runs all tasks across the web and database groups. Subsequent runs are much faster.
====

. Watch the output carefully. You should see tasks execute in the order you specified in your prompt:
+
   * Gathering Facts (automatic)
   * Install security updates (only on node1 and node2)
   * Create user padawan (on all nodes)
   * Install httpd (only on node1 and node2)
   * Ensure httpd is running (only on node1 and node2)
   * Deploy motd template (on all nodes)

. The *PLAY RECAP* at the end summarizes changes made across all hosts:
+
image::navigator-output.png[ansible-navigator run output showing task results and PLAY RECAP,link=self,window=blank,width=700,title="Playbook run output"]

[NOTE]
====
The first task in every run is *Gathering Facts*. Ansible automatically collects system information from each managed node — hostname, OS version, IP addresses, network interfaces, installed packages — before running any tasks. This fact data is what makes templates dynamic (each host renders a different MOTD based on its own hostname and OS).
====

.☑️ Task 2 - Verify user creation across all hosts
[%collapsible]
=====
[NOTE]
====
Stay in the *Control* tab.
====

The playbook created the user *padawan* on all hosts (node1, node2, and node3). Verify this worked on each node.

. Check node1:
+
[source,role="execute"]
----
ssh node1 id padawan
----
+
.Expected output
----
uid=1002(padawan) gid=1002(padawan) groups=1002(padawan)
----

. Check node2:
+
[source,role="execute"]
----
ssh node2 id padawan
----

. Check node3:
+
[source,role="execute"]
----
ssh node3 id padawan
----

All three should show the user exists. This demonstrates that tasks without a `when:` conditional ran on *all* hosts.

.☑️ Task 3 - Verify httpd is installed and running (web group only)
[%collapsible]
=====
[NOTE]
====
Stay in the *Control* tab.
====

The playbook should have installed and started Apache *only* on the web group (node1 and node2), not on the database server (node3).

. Check httpd status on node1:
+
[source,role="execute"]
----
ssh node1 systemctl is-active httpd
----
+
.Expected output
----
active
----

. Check httpd status on node2:
+
[source,role="execute"]
----
ssh node2 systemctl is-active httpd
----

. Try to check httpd on node3 (the database server):
+
[source,role="execute"]
----
ssh node3 systemctl is-active httpd 2>&1 || echo "httpd not installed (expected)"
----
+
.Expected output
----
httpd not installed (expected)
----

This proves the `when: inventory_hostname in groups['web']` conditionals worked correctly — tasks that specified this condition only ran on node1 and node2.

.☑️ Task 4 - Verify the dynamic MOTD template
[%collapsible]
=====
[NOTE]
====
Stay in the *Control* tab.
====

The playbook deployed a Jinja2 template to `/etc/motd` on all hosts. Each host should have a *different* MOTD because the template uses host-specific facts.

. View the MOTD on node1:
+
[source,role="execute"]
----
ssh node1 cat /etc/motd
----
+
.Expected output (hostname and architecture will match node1)
----
Welcome to node1.
OS: RedHat 9.4
Architecture: x86_64
----

. View the MOTD on node2:
+
[source,role="execute"]
----
ssh node2 cat /etc/motd
----
+
Notice the hostname changed to `node2` even though both nodes used the same template file.

. View the MOTD on node3:
+
[source,role="execute"]
----
ssh node3 cat /etc/motd
----

Each host rendered a unique MOTD because Ansible substituted each host's facts (`ansible_hostname`, `ansible_distribution`, `ansible_distribution_version`, `ansible_architecture`) into the template at runtime. This is why templates are powerful — one source file generates customized output for every host.

.☑️ Task 5 - Discover idempotency
[%collapsible]
=====

====
.What is idempotency?
[%collapsible]
===
*Idempotency* means that running an operation once produces the same result as running it many times. An idempotent Ansible task only makes a change when the desired state does not already exist. If the state is already correct, the task reports `ok` and does nothing.

This is what separates Ansible from shell scripts: you can safely run a playbook repeatedly and it will never break a system that is already configured correctly. Ansible checks the current state before every task and only acts when necessary.
===
====

[NOTE]
====
Switch to the *Control* tab if you aren't already there.
====

. Run the playbook a second time:
+
[source,role="execute"]
----
ansible-navigator run system_setup.yml
----

. Compare the output to the first run. Notice:

* Task status changed from *changed* (yellow) to *ok* (green)
* The *PLAY RECAP* shows `changed=0` for all hosts — Ansible confirmed everything was already in the desired state
+
image::idem-output.png[Second run showing ok status and changed=0 in PLAY RECAP,link=self,window=blank,width=700,title="Idempotent second run"]

. Switch to the *VS Code* tab, open *system_setup.yml*, select all the content, right-click, and choose *Ansible Lightspeed → Explain*. Read the explanation and look for mentions of idempotency and state checking.

[IMPORTANT]
====
Idempotency is why Ansible playbooks are safe to run repeatedly in production. Whether you run a playbook once or a hundred times, the end state is the same. This makes Ansible ideal for configuration management, drift correction, and disaster recovery — you can re-run the same playbook to fix issues without fear of breaking things.
====

== Learning Outcomes

By completing this module, you now understand:

* How to run an Ansible playbook with `ansible-navigator run`
* What Gathering Facts does and why Ansible runs it automatically
* How to verify that tasks executed correctly on the right hosts
* How conditionals control which tasks run on which inventory groups
* How Jinja2 templates render differently on each host using facts
* What idempotency means and why it makes Ansible safe for production use
* How to read a PLAY RECAP to understand what changed (or didn't change) on each run
