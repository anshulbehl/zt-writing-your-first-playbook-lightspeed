= 1.1 - Generating Your First Playbook with Lightspeed

Playbooks in Ansible are scripts written in YAML that define what tasks to run on your managed nodes. Instead of writing every line by hand, you will use *Ansible Lightspeed* to generate your first playbook from plain-English task descriptions.

In this challenge, you will generate a playbook called *system_setup.yml* that installs kernel security updates and creates a new user on a managed node.

== How Lightspeed Inline Generation Works

=====
.Lightspeed inline task generation
[%collapsible]
====
When you open an Ansible YAML file and write a task `name:` field, Lightspeed reads your description and *suggests the module and all its parameters* on the next line. Here is the workflow:

. Open or create an Ansible `.yml` file.
. Under the `tasks:` block, type a task name:
+
[source,yaml]
----
    - name: Install security updates for the kernel
----
. Press *Enter* to go to the next line. Lightspeed analyzes your description and shows a ghost-text suggestion.
. Press *Tab* to accept the suggestion. Lightspeed fills in the module and parameters for you.
. Review what was generated — make sure it matches your intent before moving on.

[TIP]
====
The better your task name, the better Lightspeed's suggestion. Be specific: instead of "install package", write "install the httpd web server package".
====
====
=====


== Hands-On Experience

======
.☑️ Task 1 - Create the system_setup.yml file
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

. In the VS Code file explorer, right-click in a blank space within the *ansible-files* directory and select *New File*.
. Name the file *system_setup.yml* and press Enter.
. Add the following play header to the file. This tells Ansible to run on *node1* with elevated privileges:
+
[source,yaml]
----
---
- name: Basic System Setup
  hosts: node1
  become: true
  tasks:
----
+
[IMPORTANT]
====
YAML is whitespace-sensitive. Use 2 spaces per indentation level — never tabs.
====
=====
======


======
.☑️ Task 2 - Use Lightspeed to generate the kernel update task
[%collapsible]
=====
[NOTE]
====
Still in *VS Code*, editing *system_setup.yml*.
====

. Position your cursor after the `tasks:` line and press Enter to start a new line.
. Type the following task name exactly (Lightspeed uses it as the prompt):
+
[source,yaml]
----
    - name: Install security updates for the kernel
----
. Press *Enter* to move to the next line. Watch for the ghost-text suggestion from Lightspeed.
. Press *Tab* to accept it.

[TIP]
====
*Expected Lightspeed output* — you should see something like:

[source,yaml]
----
    - name: Install security updates for the kernel
      ansible.builtin.dnf:
        name: 'kernel'
        state: latest
        security: true
----

Lightspeed uses the *ansible.builtin.dnf* module (the fully qualified collection name for DNF package management) and sets `security: true` to limit updates to security patches only.
====
=====
======


======
.☑️ Task 3 - Use Lightspeed to generate the user creation task
[%collapsible]
=====
. After the kernel update task, press Enter twice to add a blank line, then type:
+
[source,yaml]
----
    - name: Create a new user named myuser
----
. Press *Enter*, then *Tab* to accept Lightspeed's suggestion.

[TIP]
====
*Expected Lightspeed output:*

[source,yaml]
----
    - name: Create a new user named myuser
      ansible.builtin.user:
        name: myuser
        state: present
        create_home: true
----

Lightspeed picks the *ansible.builtin.user* module and sets sensible defaults: `state: present` ensures the user exists and `create_home: true` creates their home directory.
====

. Save the file with *Ctrl+S*.
=====
======


======
.☑️ Task 4 - Ask Lightspeed to explain the playbook
[%collapsible]
=====
Now that you have a complete playbook, let Lightspeed walk you through it.

. Select *all the content* in *system_setup.yml* (Ctrl+A).
. Right-click and choose:
+
[source,text]
----
Ansible Lightspeed → Explain
----
. Read through Lightspeed's explanation. Pay attention to:
+
* Why `become: true` is needed
* What `ansible.builtin.dnf` does differently from the bare `dnf` command
* What `state: present` means for idempotency

[NOTE]
====
Your complete *system_setup.yml* should look like this:

[source,yaml]
----
---
- name: Basic System Setup
  hosts: node1
  become: true
  tasks:
    - name: Install security updates for the kernel
      ansible.builtin.dnf:
        name: 'kernel'
        state: latest
        security: true

    - name: Create a new user named myuser
      ansible.builtin.user:
        name: myuser
        state: present
        create_home: true
----
====
