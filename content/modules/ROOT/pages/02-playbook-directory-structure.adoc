= 1.1 - Generating Your First Playbook with Lightspeed

Playbooks are the heart of Ansible automation — YAML scripts that define exactly what to do on your managed nodes. In this challenge, you will use *Ansible Lightspeed* to generate your first playbook from a plain-English prompt instead of writing YAML by hand.

The playbook will perform a basic system setup on *node1*: install kernel security updates and create a new user.

== How Lightspeed Playbook Generation Works

=====
.The generation workflow
[%collapsible]
====
Lightspeed's playbook generation works like this:

. Open the Lightspeed generation panel by clicking the *Lightspeed icon* in the VS Code Activity Bar (the vertical icon strip on the far left), or use the Command Palette (`Ctrl+Shift+P`) and search for *Ansible Lightspeed: Playbook generation*.

. A panel opens where you type a plain-English description of what you want the playbook to do.

. Lightspeed analyzes your prompt against its knowledge of Ansible modules, best practices, and the Galaxy content ecosystem, then generates a complete playbook.

. The generated playbook opens in a *new editor tab*.

. Review the YAML — check that the modules, parameters, and structure match your intent.

. Press `Ctrl+S` to save it. Name the file and choose a location inside *ansible-files*.

[TIP]
====
*Better prompts produce better playbooks.* Be specific about:

* Which hosts to target (e.g. "on node1" or "on web servers")
* What state you want (e.g. "install", "ensure running", "create")
* Any constraints (e.g. "security updates only", "create a home directory")
====

== Hands-On Experience

======
.☑️ Task 1 - Generate system_setup.yml with Lightspeed
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

. Open the Lightspeed generation panel from the Activity Bar or Command Palette.

. Enter the following prompt in the generation panel:
+
----
Write a playbook targeting node1 that installs security updates for the kernel and creates a new user named myuser with a home directory
----

. Wait for Lightspeed to generate the playbook. It opens in a new editor tab.

. Review the generated YAML. Verify that it includes:

* `hosts: node1` targeting the correct node
* `become: true` for privilege escalation
* An `ansible.builtin.dnf` task with `security: true` for the kernel update
* An `ansible.builtin.user` task creating `myuser` with `create_home: true`

[TIP]
====
*Expected Lightspeed output:*

[source,yaml]
----
---
- name: Basic System Setup
  hosts: node1
  become: true
  tasks:
    - name: Install security updates for the kernel
      ansible.builtin.dnf:
        name: 'kernel'
        state: latest
        security: true

    - name: Create a new user named myuser
      ansible.builtin.user:
        name: myuser
        state: present
        create_home: true
----

Notice that Lightspeed uses *fully qualified collection names* (`ansible.builtin.dnf`, `ansible.builtin.user`) — this is the recommended practice for all modern Ansible content.
====

. If the output looks correct, press `Ctrl+S` and save the file as *system_setup.yml* inside the *ansible-files* directory.
=====
======


======
.☑️ Task 2 - Ask Lightspeed to explain the playbook
[%collapsible]
=====
Before running the playbook, use Lightspeed to understand what was generated.

. In the new editor tab containing *system_setup.yml*, select all the content (`Ctrl+A`).
. Right-click and choose:
+
----
Ansible Lightspeed → Explain
----
. Read through the explanation. Pay attention to:

* Why `become: true` is needed for package and user management
* What `state: latest` with `security: true` does differently from a full system upgrade
* What `state: present` means for the user task and why it is idempotent
=====
======


== Learning Outcomes

By completing this module, you now understand:

* How to use Lightspeed's generation panel to produce a complete playbook from a prompt
* Why fully qualified collection names (`ansible.builtin.*`) are used in modern Ansible
* How `become: true` enables privilege escalation in a play
* What idempotency means at the task level (state-based, not action-based)
