= 1.3 - From One to Many

The real power of automation is managing *many systems at once*. In this challenge, you will use Ansible Lightspeed to regenerate the playbook so it targets the entire *web* group — both *node1* and *node2* — instead of a single host.

== Hands-On Experience

======
.☑️ Task 1 - Generate an updated playbook targeting the web group
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

. Open the Lightspeed generation panel from the Activity Bar or Command Palette.
. Enter the following prompt:
+
----
Write a playbook targeting the web group that installs security updates for the kernel and creates a user named myuser with a home directory
----
. Review the generated playbook in the new editor tab. Verify that:

* `hosts:` is set to `web` (not `node1`)
* `become: true` is present
* Both tasks from the previous module are included

[TIP]
====
*Expected Lightspeed output:*

[source,yaml]
----
---
- name: Basic System Setup
  hosts: web
  become: true
  tasks:
    - name: Install security updates for the kernel
      ansible.builtin.dnf:
        name: 'kernel'
        state: latest
        security: true

    - name: Create a new user named myuser
      ansible.builtin.user:
        name: myuser
        state: present
        create_home: true
----

By changing the prompt from "targeting node1" to "targeting the web group", Lightspeed generates `hosts: web`. This single change makes the playbook run against every host in the `[web]` inventory group — currently node1 and node2, but automatically any future web hosts you add.
====

. Press `Ctrl+S` and save over the existing *system_setup.yml* in the *ansible-files* directory.
=====
======


======
.☑️ Task 2 - Run the updated playbook
[%collapsible]
=====
[NOTE]
====
Switch to the *Control* tab.
====

[source,role="execute"]
----
cd /home/rhel/ansible-files
----

[source,role="execute"]
----
ansible-navigator run system_setup.yml
----

[WARNING]
====
This run may take a minute or two — *node2* is being configured for the first time, so it needs to download metadata and run the kernel update task.
====
=====
======


======
.☑️ Task 3 - Verify myuser exists on both nodes
[%collapsible]
=====
. Check *node1*:
+
[source,role="execute"]
----
ssh node1 id myuser
----

. Check *node2*:
+
[source,role="execute"]
----
ssh node2 id myuser
----
+
.Expected output (both nodes)
----
uid=1002(myuser) gid=1002(myuser) groups=1002(myuser)
----

[NOTE]
====
If you add more web servers to the inventory later, you do not need to change the playbook. Setting `hosts: web` means every host in that group is automatically included in future runs.
====
=====
======


== Learning Outcomes

By completing this module, you now understand:

* How to use inventory groups to target multiple hosts with a single playbook
* Why `hosts: web` is more scalable than `hosts: node1`
* How Lightspeed adapts its output based on the group or host name in your prompt
* How ansible-navigator manages parallel execution across multiple nodes
