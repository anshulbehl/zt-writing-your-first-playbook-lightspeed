.What is Ansible Lightspeed?
[%collapsible]
====
*Ansible Lightspeed* is an AI-powered coding assistant built into the *Red Hat Ansible* VS Code extension. It helps you write Ansible automation faster and more accurately by generating playbook tasks from plain-English descriptions.

Here is what Lightspeed can do for you:

* *Inline task generation* — type a task name, press Enter, and Lightspeed suggests the module and all its parameters.
* *Explain* — select any block of YAML, right-click, and ask Lightspeed to explain what it does.
* *Content source tracing* — Lightspeed tells you which Ansible Galaxy content informed its suggestion, so you know it is grounded in real automation patterns.

In this lab, you will use Lightspeed to generate every playbook. Your job is to write good prompts, review what the AI produces, and run the result.

== Hands-On Experience

======
.☑️ Task 1 - Verify Lightspeed is connected
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab at the top of the screen.
====

. Look at the bottom status bar of VS Code. You should see the Ansible Lightspeed icon — it looks like a small lightning bolt or the Ansible logo. A green indicator means Lightspeed is connected and ready.

. Open the *Command Palette* with `Ctrl+Shift+P` (or `Cmd+Shift+P` on Mac) and type:
+
[source,text]
----
Ansible: Connect to Lightspeed
----
+
Select the option and confirm that the connection succeeds. You should see a notification in the bottom-right corner of VS Code confirming the connection.

[TIP]
====
Lightspeed is pre-configured in this lab to use a local proxy. You do not need to log in or provide any credentials.
====

======
.☑️ Task 2 - Explore the pre-created inventory
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

Your lab environment already has an inventory file ready. An *inventory* file tells Ansible which nodes to manage. It organizes hosts into groups so you can target specific systems with your playbooks.

. In the VS Code file explorer on the left, open the *ansible-files* directory. You will find a file named *inventory*. Click it to open it.
+
image::vscode_inventory.png[Inventory, 90%, align="center"]

. The file contains the following:
+
[source,ini]
----
[web]
node1
node2

[database]
node3

[nodes:children]
web
database
----

. Now let Lightspeed explain it for you. *Select all the text* in the inventory file (Ctrl+A), then *right-click* and choose:
+
[source,text]
----
Ansible Lightspeed → Explain
----
+
Read through Lightspeed's explanation. It will describe what each group does and why organizing hosts into groups is powerful for automation.
=====
======


======
.☑️ Task 3 - Use ansible-navigator to inspect the inventory
[%collapsible]
=====
[NOTE]
====
Switch to the *Control* tab.
====

. Change into the *ansible-files* directory:
+
[source,shell]
----
cd /home/rhel/ansible-files
----

. Run the following command to list all hosts and groups in the inventory:
+
[source,shell]
----
ansible-navigator inventory --list
----
+
.The output should look similar to this:
[source,json]
----
{
    "_meta": {
        "hostvars": {}
    },
    "all": {
        "children": [
            "ungrouped",
            "web",
            "database",
            "nodes"
        ]
    },
    "database": {
        "hosts": ["node3"]
    },
    "web": {
        "hosts": ["node1", "node2"]
    }
}
----

. You can also view a condensed graph of the inventory:
+
[source,shell]
----
ansible-navigator inventory --graph
----

[NOTE]
====
*ansible-navigator* is the command-line tool you will use throughout this lab to run your Ansible playbooks. It launches an Execution Environment (a container) that includes all the collections and dependencies you need.
