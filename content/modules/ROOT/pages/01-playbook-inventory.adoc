.What is Ansible Lightspeed?
[%collapsible]
====
*Ansible Lightspeed* is an AI-powered coding assistant built into the *Red Hat Ansible* VS Code extension. It generates complete, ready-to-run Ansible playbooks from plain-English descriptions — no YAML knowledge required to get started.

Here is what Lightspeed can do in this lab:

* *Playbook generation* — describe what you want in plain English, and Lightspeed produces a complete playbook in a new editor tab that you can save and run immediately.

* *Explain* — select any block of YAML, right-click, and ask Lightspeed to explain what it does in plain English.

* *Content source tracing* — Lightspeed cites which Ansible Galaxy content informed its suggestion, so every generated task is grounded in real-world automation patterns.

In this lab, *you will use Lightspeed to generate every playbook*. Your job is to write clear prompts, review what Lightspeed produces, save the result, and run it.

== Hands-On Experience

======
.☑️ Task 1 - Verify Lightspeed is connected
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab at the top of the screen.
====

. Look at the bottom *status bar* of VS Code. You should see the Ansible Lightspeed icon — a small AI or lightning indicator. Green means connected and ready.

. Open the *Command Palette* with `Ctrl+Shift+P` and type:
+
[source,role="execute"]
----
Ansible: Connect to Lightspeed
----

Select the option and confirm the connection succeeds. A notification in the bottom-right corner of VS Code confirms it.

[TIP]
====
Lightspeed is pre-configured in this lab to use a local AI proxy. You do not need to log in or provide any credentials.
====
=====
======


======
.☑️ Task 2 - Explore the pre-created inventory
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

Your lab environment already has an inventory file ready. An *inventory* tells Ansible which nodes to manage and organizes them into groups so you can target specific systems.

. In the VS Code file explorer on the left, open the *ansible-files* directory and click the *inventory* file to open it.
+
image::vscode_inventory.png[Inventory file open in VS Code editor,link=self,window=blank,width=700,title="Pre-created inventory file"]

. The file contains:
+
[source,ini]
----
[web]
node1
node2

[database]
node3

[nodes:children]
web
database
----

. Let Lightspeed explain it. *Select all the text* in the file (`Ctrl+A`), then *right-click* and choose:
+
----
Ansible Lightspeed → Explain
----

Read through Lightspeed's explanation. Notice how it describes the group structure and why grouping hosts matters for targeted automation.
=====
======


======
.☑️ Task 3 - Inspect the inventory with ansible-navigator
[%collapsible]
=====
[NOTE]
====
Switch to the *Control* tab.
====

. Change into the *ansible-files* directory:
+
[source,role="execute"]
----
cd /home/rhel/ansible-files
----

. List all hosts and groups:
+
[source,role="execute"]
----
ansible-navigator inventory --list
----
+
.Expected output
----
{
    "_meta": { "hostvars": {} },
    "all": {
        "children": ["ungrouped", "web", "database", "nodes"]
    },
    "database": { "hosts": ["node3"] },
    "web": { "hosts": ["node1", "node2"] }
}
----

. View a condensed graph of the inventory:
+
[source,role="execute"]
----
ansible-navigator inventory --graph
----

[NOTE]
====
*ansible-navigator* runs playbooks inside a container called an *Execution Environment* (EE). The EE includes all the Ansible collections and dependencies needed for this lab — you do not need to install anything manually.
====
