= 1.8 - Templates

Jinja2 templates let you generate *dynamic configuration files* that are customized for each managed node at runtime. Instead of copying a static file, Ansible fills in variables like hostname, IP address, or OS version before placing the file on the target.

In this challenge, you will use Ansible Lightspeed to generate a Jinja2 template for `/etc/motd` (the message of the day) and the playbook task that deploys it.

=====
.How templates work in Ansible
[%collapsible]
====
* Template files use the `.j2` extension and contain a mix of static text and `{{ variable }}` placeholders.
* The `ansible.builtin.template` module renders the template on the control node (substituting variables with facts gathered from each target host) and copies the result to the managed node.
* Because Ansible facts are host-specific, the same template produces a different file on each managed node.
====
=====

== Hands-On Experience

======
.☑️ Task 1 - Generate the Jinja2 template with Lightspeed
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

. In the *ansible-files* directory, right-click and create a new folder named *templates*.
. Inside *templates*, right-click and create a new file named *motd.j2*.
. The file is a `.j2` template, not a playbook — but you can still ask Lightspeed to help. Type the following in the file:
+
[source,jinja2]
----
Welcome to {{ ansible_hostname }}.
----
. Let Lightspeed suggest additional lines. Place your cursor at the end of the line, press Enter, and continue typing:
+
[source,text]
----
OS:
----
+
Watch for Lightspeed to suggest the OS fact variable. Press Tab to accept, then continue with a new line for Architecture.

[TIP]
====
*Expected completed motd.j2:*

[source,jinja2]
----
Welcome to {{ ansible_hostname }}.
OS: {{ ansible_distribution }} {{ ansible_distribution_version }}
Architecture: {{ ansible_architecture }}
----

`ansible_hostname`, `ansible_distribution`, `ansible_distribution_version`, and `ansible_architecture` are all *Ansible facts* — they are populated automatically by the *Gathering Facts* task at the start of every play.
====

. Save the file.
=====
======


======
.☑️ Task 2 - Use Lightspeed to add the template task to the playbook
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab and open *system_setup.yml*.
====

. Before the `handlers:` section, add a new task. Type:
+
[source,yaml]
----
    - name: Deploy MOTD from Jinja2 template to all nodes
----
. Press *Enter*, then *Tab* to accept Lightspeed's suggestion.

[TIP]
====
*Expected Lightspeed output:*

[source,yaml]
----
    - name: Deploy MOTD from Jinja2 template to all nodes
      ansible.builtin.template:
        src: templates/motd.j2
        dest: /etc/motd
----

Lightspeed recognizes "Jinja2 template" in the task name, picks the correct module, and resolves the source path relative to the playbook directory.
====

. Save the file.
=====
======

[NOTE]
====
Switch to the *Control* tab.
====

cd /home/rhel/ansible-files

[source,shell]
----
ansible-navigator run system_setup.yml

======
.☑️ Task 4 - Verify the MOTD on each node
[%collapsible]
=====
. SSH into *node1* and check the MOTD:
+
[source,shell]
----
ssh node1
----
+
.You should see something like (version numbers will vary):
[source,text]
----
Welcome to node1.
OS: RedHat 9.2
Architecture: x86_64
----
. Exit and try *node2* — notice the hostname is different:
+
[source,shell]
----
exit
----
+
[source,shell]
----
ssh node2
----
. Exit node2:
+
[source,shell]
----
exit
----

[NOTE]
====
The same `motd.j2` template produced a *unique file* on each node because Ansible substituted the host-specific facts at runtime. This is the power of templating.
====
