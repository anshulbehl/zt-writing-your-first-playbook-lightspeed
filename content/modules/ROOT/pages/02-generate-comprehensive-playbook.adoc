= 1.1 - Generate a Comprehensive Playbook with the Coding Assistant

Real-world playbooks handle multiple concerns in a single automation run — user management, package installation, service configuration, and dynamic file deployment. In this challenge, you will use the *Red Hat Ansible VS Code extension's Automation Coding Assistant* to generate a complete, production-ready playbook from a single well-structured prompt.

Instead of writing YAML by hand or iterating through multiple attempts, you will learn how to construct a *comprehensive prompt* that gives the coding assistant all the context it needs to generate exactly what you want in one shot.

== Understanding Comprehensive Prompts

=====
.💡 What makes a good LLM prompt for playbook generation?
[%collapsible]
====
When prompting the coding assistant to generate a playbook, *specificity and structure* produce better results than vague requests. A comprehensive prompt should include:

* *Target hosts* — which inventory group or specific hosts
* *Required privilege escalation* — does it need `become: true`?
* *Variables to define* — reusable values like usernames, package names, service names
* *Tasks in logical order* — what to install, configure, start, and verify
* *Conditionals* — which tasks run only on specific host groups
* *Loops* — repeating tasks across multiple items (users, packages, files)
* *Templates* — dynamic configuration files that use Jinja2 and host facts

The more detail you provide, the more accurate Lightspeed's output will be.
====
=====


== Hands-On Experience

======
.☑️ Task 1 - Construct the comprehensive prompt
[%collapsible]
=====
[NOTE]
====
Switch to the *VS Code* tab.
====

You will construct a prompt that asks the coding assistant to generate a playbook covering all the essentials. Copy and customize the template below, filling in the details where indicated.

Your prompt should request a playbook that:

. Targets the *`all`* hosts (includes both web and database groups)
. Uses privilege escalation (`become: true`)
. Defines these variables:
   * `user_name` — set to `padawan`
   * `package_name` — set to `httpd`
   * `apache_service_name` — set to `httpd`
. Performs these tasks *in order*:
   .. Install security updates for the kernel package (only on the `web` group)
   .. Create the user defined in `user_name` with a home directory
   .. Install the package defined in `package_name` (only on the `web` group)
   .. Ensure the Apache service is running and enabled (only on the `web` group)
   .. Deploy a Jinja2 template from `templates/motd.j2` to `/etc/motd` on all hosts

[TIP]
====
When specifying conditionals, tell the coding assistant to use `when: inventory_hostname in groups['web']` to target only the web servers.
====

*Use this prompt* (copy it exactly):

----
Create an Ansible playbook named "Basic System Setup" that targets all hosts and uses privilege escalation.

Define variables for user_name (padawan), package_name (httpd), and apache_service_name (httpd).

Include these tasks in order:
1. Install security updates for the kernel package, but only on hosts in the web group using the conditional: when: inventory_hostname in groups['web']
2. Create a user with the name from user_name variable, ensuring a home directory is created
3. Install the package from package_name variable, only on the web group using the same conditional
4. Ensure the Apache service (from apache_service_name) is running and enabled, only on the web group using the same conditional
5. Deploy a Jinja2 template from templates/motd.j2 to /etc/motd on all hosts

Include a handler to restart Apache when needed, and use ansible.builtin modules throughout.
----

[IMPORTANT]
====
Copy the prompt above exactly as shown. The coding assistant works best with specific, detailed prompts that include:

* The exact conditional syntax to use (`when: inventory_hostname in groups['web']`)
* Module family preference (`ansible.builtin` modules)
* Handler requirements

This ensures the generated playbook matches what the validation system expects.
====

======
.☑️ Task 2 - Generate the playbook with the coding assistant
[%collapsible]
=====
[NOTE]
====
Stay in the *VS Code* tab.
====

. Click on the *Ansible extension icon* in the left sidebar to open the Ansible extension panel.

. Click the *"Generate a Playbook with Ansible"* button in the extension panel.

. When prompted, paste your comprehensive prompt from Task 1.

. The coding assistant will generate the playbook and show you a preview with suggested tasks. *Review the suggested steps* to confirm they match your prompt, then click *"Create Playbook"* to manually confirm and create the file.

. The playbook will open in a new tab. Review the output to confirm it includes:
+
   * A `vars:` section with `user_name`, `package_name`, and `apache_service_name`
   * Tasks with `when:` conditionals targeting the `web` group
   * A `template:` task referencing `templates/motd.j2`
   * A `handlers:` section for restarting Apache

. Save the file as `system_setup.yml` in the `/home/rhel/ansible-files/` directory:
+
   * Press `Ctrl+S`
   * Navigate to `/home/rhel/ansible-files/`
   * Name the file `system_setup.yml`
   * Click *Save*

[NOTE]
====
If the coding assistant's output is missing a required section (vars, handlers, conditionals, or template task), you can click the image:solve.png[Solve, 55] button to get a working version, then compare it to what you generated to understand what was different.
====
=====

======
.☑️ Task 3 - Use the coding assistant to explain the generated playbook
[%collapsible]
=====
[NOTE]
====
Stay in the *VS Code* tab with `system_setup.yml` open.
====

Now that you have a complete playbook, use the coding assistant's *Explain* feature to understand it.

. Make sure your `system_setup.yml` file is open in the editor.

. Click on the *Ansible extension icon* in the left sidebar to open the Ansible extension panel.

. Click the *"Explain Playbook"* button in the extension panel.

. The coding assistant will analyze your entire playbook and provide an explanation covering:
   * What the playbook does overall
   * How the `vars:` section defines reusable values
   * How `when:` conditionals target specific inventory groups
   * How the `template:` task deploys dynamic configuration files
   * How `handlers:` work and when they run

[NOTE]
====
The *Explain Playbook* feature is how you learn from the coding assistant's output. It provides a comprehensive breakdown of what your automation does and why each section matters. You can use this feature on any playbook you encounter to build your Ansible knowledge without leaving the editor.
====
