## TL;DR for the agent

The block is **not** on either VM's OS. Both VMs' OS firewalls are fully open
and `sshd` is listening on `0.0.0.0:22`. The drop happens at the **cluster
network layer (Kubernetes NetworkPolicy)**, which is generated by the platform
from `config/firewall.yaml`.

The single most likely missing rule is **egress TCP/22 on the control pod**.
The repo's `firewall.yaml` declares an explicit `egress:` allowlist of `{80, 443}`,
which most likely *replaced* the platform default that previously permitted
VM-to-VM SSH. Result: control physically cannot open an outbound port-22
connection. Add 22 (and 53) back to egress, and add 22 to ingress.

**Do step 0 (confirm) if you have `oc`, then apply Fix 1. Then do the cleanups
in Fix 2–4.**

## Confirmed facts (live evidence)

| Check | Result | Implication |
|---|---|---|
| `firewall-cmd` on node01 | `command not found` | **No firewalld on nodes.** All node firewalld scripts are pointless. |
| `ss -tlnp \| grep :22` on node01 | listening on `0.0.0.0:22` and `[::]:22` | sshd is up and bound on all interfaces. |
| `firewall-cmd --list-all` on control | zone `trusted`, target `ACCEPT` | Control OS firewall is wide open. |
| `ip route` on control | `src 10.0.2.2` | Control is masquerade-bound (NAT). |
| `ip route` on node01 | `src 10.0.2.2` | **Nodes are ALSO masquerade-bound** — same internal IP as control. |
| Showroom pod can SSH to nodes (runs setup) | works | Port 22 *does* reach the node guest; node sshd works. Only the **source pod** differs between working (showroom) and failing (control) → NetworkPolicy. |

### KubeVirt masquerade facts that constrain the design
- Every VM sees itself as `10.0.2.2` behind its **own** pod's NAT. The
  `10.130.x.x` addresses are the **pod IPs** of each node's launcher pod.
- You reach a node **only via its pod IP** (or the pod DNS name that resolves to
  it), never via `10.0.2.x`.
- If a masquerade interface's `ports:` list is **omitted**, all ports forward
  into the guest; if it is **specified**, only those listed ports forward.
- ICMP to the pod IP is generally **not** forwarded into the guest. **Ping is not
  a valid test here.** Use `nc -vz <pod-ip> 22`.

## What was ruled out (do not revisit)

- ❌ "firewalld on nodes blocks SSH" — there is no firewalld on the nodes.
- ❌ "node OS firewall / control OS firewall" — both are fully permissive.
- ❌ "Isolated network has only one IP" — there is no isolated network; `10.0.2.2`
  is a per-pod NAT address shared by every VM.
- ❌ "Service type (SSH vs HTTP) determines subnet placement" — it does not;
  every VM is masquerade-bound regardless.
- ❌ "Add a static route to 10.130.0.0/16" — redundant with the masquerade
  default route; does nothing. Remove it.
- ❌ "Bidirectional routing / return route missing on nodes" — not the cause;
  masquerade SNAT handles return traffic via conntrack.

## Fix 1 — Patch `config/firewall.yaml` (PRIMARY FIX)

Add **egress 22** (the actual missing rule), **ingress 22**, and re-declare DNS
so a replace-semantics platform does not break name resolution. Listing
everything explicitly is correct whether the platform merges or replaces.

```yaml
egress:
  - ports:
      - { protocol: TCP, port: 443 }
      - { protocol: TCP, port: 80 }
      - { protocol: TCP, port: 22 }    # control -> nodes SSH (the missing rule)
      - { protocol: UDP, port: 53 }    # keep cluster DNS working
      - { protocol: TCP, port: 53 }

ingress:
  - ports:
      - { protocol: TCP, port: 8080 }
      - { protocol: TCP, port: 22 }    # inbound SSH to nodes
```

Notes:
- If `firewall.yaml` supports peer scoping (namespace/pod selectors), scope the
  ingress-22 rule to the lab namespace or the control pod instead of allowing
  port 22 from anywhere. Port-only is acceptable for a lab.
- After provisioning, verify from control: `nc -vz <node01-pod-ip> 22` then
  `ssh rhel@node01`.

## Fix 3 — Remove the static-route hack

In `setup-automation/setup-control.sh`, delete the block that adds:

```bash
ip route add 10.130.0.0/16 via 10.0.2.1 dev eth0
```

It is redundant with the existing masquerade default route and reinforces the
wrong "two subnets" model.

---

## Fix 4 — Inventory must use pod DNS names / pod IPs

Ensure the Ansible inventory and any `/etc/hosts` population target the **pod DNS
names** (e.g. `node01.lab.<sandbox>.svc.cluster.local`) or pod IPs
(`10.130.x.x`). **Never** map a node to `10.0.2.x` — that is each VM's own NAT
loopback and will resolve to the local VM.

If a setup step builds `/etc/hosts` via `getent hosts`, verify it records
distinct pod IPs per node (the earlier "duplicate IPs" symptom is consistent
with something resolving to `10.0.2.2`).

## DO NOT DO — rejected approaches

- ❌ **Change node services from SSH/22 to HTTP/80** (old "Option 1").
  If the platform builds the masquerade interface `ports:` from the service,
  switching to 80 makes the node forward **only 80** into the guest and **breaks
  SSH into the node guest** — including the showroom setup path that currently
  works. This is a regression, not a fix.
- ❌ Re-adding firewalld/iptables logic to node setup scripts.
- ❌ Re-adding the `10.130.0.0/16` static route.
- ❌ Treating `ping` failure as a connectivity signal (masquerade doesn't forward
  ICMP to the guest). Use `nc -vz <pod-ip> 22`.
